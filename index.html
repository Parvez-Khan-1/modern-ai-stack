01 · The Gap

The Gap Nobody Talks About Openly

The space between where education ends and industry begins

How many of you have trained a machine learning model? Probably most of you. Now — how many of you have deployed one to a million real users? That silence is exactly where today's session lives.

There is a natural and necessary gap between what education prepares you for and what industry demands on day one — and not enough people talk about it openly. It is not a flaw in how you are being taught. It is simply the reality that building things at scale introduces a completely different set of engineering problems that only become visible once you are in the middle of them.

The reality of production AI

In a live system, your model is rarely the hardest problem. The data pipelines, the latency, the hallucinations, the cost, the monitoring — that is where engineers spend most of their time.

I've spent 10 years on the production side of this. At Mastercard, I've seen brilliant models fail spectacularly in production — not because the model was wrong, but because everything around it wasn't engineered to survive contact with reality.

AI in 2025 is no longer a model sitting in a notebook. It is a living system — with layers of data, orchestration, retrieval, safety, and infrastructure all working in coordination. The engineers who understand how these layers connect are the ones companies are desperately hiring for. Understanding the stack is what separates an AI user from an AI engineer.

02 · The Evolution

The AI Evolution — From Rules to Agents

Every layer of the Modern AI Stack exists because a previous era hit a wall

The Modern AI Stack didn't appear overnight. Each layer — the gateway, the orchestrator, the vector database, the model — was added to solve a specific problem that the previous generation of AI simply couldn't handle. Understanding this progression tells you why each layer exists, not just what it does.

1980s – 1990s
Rule-Based AI

Hand-crafted rules, expert systems, decision trees. If this then that. Brittle, expensive to maintain, impossible to scale. The wall it hit: couldn't handle ambiguity or learn from new data.

Expert SystemsDecision TreesLISP / Prolog → Led to: ML models that learn
2000s – 2012
Classical Machine Learning

Models learned patterns from data. Feature engineering was everything — humans still decided what the model should look at. The wall it hit: couldn't process raw unstructured data like images, text, or audio.

SVMRandom ForestXGBoostFeature Engineering → Led to: deep learning on raw data
2012 – 2017
Deep Learning Revolution

Neural networks learned their own features. GPUs unlocked scale. Vision and speech were transformed. The wall it hit: each model was task-specific — you needed a separate model for every single problem.

CNNsRNNsAlexNetTensorFlow → Led to: foundation models trained once, used everywhere
2017 – 2022
The Transformer Era

"Attention is All You Need" changed everything. LLMs could understand context at scale — BERT, GPT-2, the foundations of modern NLP. The wall it hit: LLMs were trained on public data and knew nothing about your private, real-time organisational knowledge.

TransformersBERTGPT-2Attention → Led to: RAG & vector databases
2022 – 2024
LLMs Go Public + RAG Emerges

ChatGPT reached 100 million users in 60 days. RAG solved the private data problem. But LLMs could still only answer questions — they couldn't take actions, use tools, or complete multi-step tasks autonomously. The wall it hit: passive intelligence isn't enough for real enterprise workflows.

GPT-4ClaudeRAGPrompt Engineering → Led to: Agentic AI
2024 – Present
Agentic AI & The Full Stack We are here

AI moves from answering questions to taking actions. Multi-agent systems, orchestration layers, RAG pipelines, safety gateways, and observability systems all become essential engineering infrastructure. This is what the Modern AI Stack is. Every layer we'll discuss today exists because of this evolution.

Agentic AI Multi-Agent Systems RAG Orchestration Production AI Stacks

Before we name each layer technically, let's install the right mental model. Because once this analogy clicks, every technical term that follows will make immediate sense.

Stop thinking of an AI system as one magical brain. Think of it as a coordinated kitchen — multiple specialists, each doing exactly one thing well, in a precise sequence.

A restaurant fails not because the chef is bad — it fails because the waiter took the wrong order, the pantry ran out of ingredients, or the manager lost track of a table. AI systems fail exactly the same way.

🔐 Gateway Layer
The security and routing boundary. Every request is validated, filtered, and directed before it reaches any AI component.

The Waiter ＋
What it does

Acts as the front door for every AI request. Authenticates the caller, enforces rate limits, scans prompts for unsafe content, and routes traffic to the right downstream service. In enterprise systems, this layer also logs every request for audit trails.

Tools & Technologies
Azure API ManagementKong GatewayAWS API GatewayNGINXPortkeyLiteLLM
Real-world example

At Mastercard, every AI service request passes through a semantic gateway that validates the caller's permissions, checks the prompt against compliance rules, and routes to either a cloud LLM or an on-premise model depending on data classification.

What breaks here

Missing rate limits lead to cost explosions. No prompt sanitisation leads to injection attacks. Poor routing logic sends sensitive data to external models when it should stay on-premise.

🎯 Orchestration Layer
The brain of coordination. Decides which model to call, which data to fetch, and in what order — especially critical for multi-step and agentic workflows.
05
Orchestration Layer
Coordinates which model, which tools, which data sources to involve
Kitchen Manager ＋
What it does

Manages the flow of a complex AI task. For a simple Q&A, it routes directly to the model. For an agentic task — "analyse this document, search the web, and write a report" — it plans and sequences multiple tool calls, handles retries, and assembles the final output.

Tools & Technologies
LangChainLlamaIndexAutoGenCrewAISemantic KernelDSPy
Real-world example

A clinical documentation AI that, upon receiving a doctor's voice note, simultaneously triggers a medical NER model, queries the patient's RAG store, validates against HIPAA rules, and formats the final structured note — all orchestrated in sequence.

What breaks here

Infinite loops in agentic tasks, no timeout handling, poor error propagation between tool calls, and context window overflow when passing too much state between steps.

🗄️ Data Layer
Where private, real-time, and proprietary knowledge lives. RAG operates here — bridging the gap between what the model was trained on and what your organisation actually knows today.
04
Retrieval Layer (RAG)
Fetches relevant private or real-time data before the model sees the query
The Pantry ＋
What it does

Converts your private documents into vector embeddings and stores them in a vector database. At query time, it finds the most semantically relevant chunks and injects them into the prompt — giving the LLM grounded, organisation-specific context it was never trained on.

Tools & Technologies
PineconeWeaviateChromaAzure AI SearchFAISSpgvectorOpenAI Embeddingssentence-transformers
Real-world example

An internal Mastercard AI assistant that can answer questions about internal policies updated yesterday — without any model retraining. The policy PDFs live in the vector store; the LLM retrieves and reasons over them at query time.

What breaks here

Poor chunking strategy (too large = noisy context, too small = lost context), stale embeddings after document updates, and retrieval returning semantically similar but factually wrong chunks — leading to confident wrong answers.

🧠 Model Layer
The intelligence engine. Just one layer of seven — yet often mistakenly treated as the entire system. The model generates responses but depends entirely on every layer above it to receive the right input.
03
LLM / SLM
The language model generating the response — the Chef, not the whole kitchen
The Chef ＋
What it does

Receives the assembled prompt — user query + retrieved context + system instructions — and generates a response. The choice between an LLM and SLM is a pure engineering tradeoff: accuracy and capability vs. cost, latency, and deployability.

Tools & Technologies
GPT-4oClaude 3.5Gemini ProLlama 3MistralPhi-3Azure OpenAIOllama
Real-world example

For Mastercard's customer-facing chatbot handling 10M+ queries per day, we use a fine-tuned SLM for intent classification (fast, cheap, accurate for known patterns) and escalate to a full LLM only for complex, unrecognised queries.

What breaks here

Hallucination when context is missing or contradictory. Context window overflow on long conversations. Model latency causing timeout cascades. And using a sledgehammer (GPT-4) where a scalpel (Phi-3) would do the job 10x cheaper.

📊 Observability & Infrastructure Layer

The physical and logical foundation of the entire stack. Manages where data is stored, how compute is allocated, how models are served at scale, and how data flows between components. This layer determines the cost and scalability ceiling of the entire system.

In the sections that follow, we go deep on two of these layers — Layer 03 (LLMs & SLMs) and Layer 04 (RAG) — because they sit at the heart of every modern AI product. Everything else in the stack exists to support, route, scale, and monitor these two.

05 · LLMs & SLMs

LLMs and SLMs — What They Actually Are

No math. Just the right mental model for an engineer.

A Large Language Model is, at its core, a very well-read autocomplete. It has read a significant portion of the internet, learned the statistical patterns of human language, and became extraordinarily good at predicting what word comes next — with context, at scale, and with surprising coherence.

But here is the critical limitation: LLMs are frozen in time. They know nothing about your organisation's internal documents, nothing about what happened last week, and nothing about private data they have never seen. This is where the LLM vs SLM distinction becomes a real engineering decision — not an academic one.

LLM — Large Language Model
Massive, powerful, expensive. GPT-4, Claude, Gemini. Billions of parameters. High accuracy, higher latency, higher cost per call.
→
Best for
Complex reasoning, nuanced understanding, creative tasks. When accuracy matters more than speed or cost.
SLM — Small Language Model
Compact, fast, deployable anywhere. Phi-3, Mistral 7B, Llama 3.2. Runs on a laptop or an edge device.
→
Best for
Latency-sensitive tasks, edge deployment, cost at scale. When you need 10 million responses per day within a budget.

At Mastercard, this is a real conversation we have for every AI feature: does this need a large model, or can a smaller, fine-tuned model do the job cheaper and faster? The answer changes everything — cost, latency, infrastructure footprint, and compliance posture.

Prompt Engineering — how you talk to the model

You can't discuss the model layer without mentioning how you communicate with it. Prompt Engineering is the discipline of crafting inputs that reliably produce the outputs you need. A poorly structured prompt to a powerful LLM gives worse results than a well-structured prompt to a smaller one. In production systems, prompts are version-controlled, tested, and monitored just like code. Think of it as the API contract between your application and the model.

06 · RAG & Agentic RAG

RAG — How Industry Makes LLMs Work with Real Data

Retrieval Augmented Generation — and how it evolves into Agentic RAG

LLMs are powerful but frozen. The naive answer is fine-tuning — retrain the model on your data. This sounds correct but is almost always the wrong engineering decision. It's expensive, slow, and in domains like banking or healthcare where data changes continuously, it simply doesn't scale.

The industry answer is Retrieval Augmented Generation — RAG:

R
Retrieve

Search your private data store for the most relevant chunks related to the user's query — semantically, not just by keyword

A
Augment

Attach those retrieved chunks to the prompt — hand the LLM a precise technical manual before asking it to answer

G
Generate

The LLM answers using only the provided context — grounded, accurate, and specific to your organisation's data

Think of it like an open-book exam. Without RAG, the LLM sits the exam with only what it memorised during training. With RAG, you hand it the exact chapter it needs — right before it answers.

The engineering insight

The LLM is not the hard part of a RAG system. Finding, cleaning, chunking, embedding, and retrieving the right data — that is where real engineering happens. Strong fundamentals matter here.

Problem
"The AI is hallucinating — giving confident answers that are factually wrong"
→
Solution
RAG — constrain the model to answer only from retrieved, verified context. No context, no answer.
Problem
"The AI doesn't know about our latest policy update from last week"
→
Solution
Update the vector database — no model retraining needed. Knowledge and model are decoupled by design.
Problem
"The AI response takes 8 seconds — users are dropping off"
→
Solution
Use a smaller model for retrieval scoring, cache frequent queries, optimise chunk sizes and retrieval depth.

Agentic RAG — When RAG Isn't Enough

Basic RAG answers one question. Agentic RAG completes a mission.

Standard RAG is a single round-trip — one query, one retrieval, one answer. But real enterprise tasks are rarely that simple. "Analyse this contract, compare it against our policy database, identify risk clauses, and draft a summary for legal review" — that is not one retrieval. That is a workflow.

Agentic RAG wraps the RAG pipeline inside an agent reasoning loop. The agent decides what to retrieve, evaluates whether it has enough information, retrieves more if needed, and only generates a final answer when it is satisfied with the evidence.

Real example — Clinical AI at scale

An oncology report agent at a hospital doesn't just answer "what is this patient's diagnosis?" It retrieves the lab report, cross-references the radiology findings, checks against clinical guidelines, identifies staging criteria, and produces a structured summary — all through an Agentic RAG loop before a single word is generated.

07 · Agentic AI & MCP

From Knowledge to Action — Agents & the MCP Protocol

RAG gives the LLM knowledge. Agents give it the ability to act. MCP gives it a standardised way to connect to anything.

RAG solved one problem — an LLM that knows your data. But there is a second, deeper problem: knowing is not the same as doing. An LLM with RAG can tell you the answer. An Agent can go and do something about it.

The fundamental shift

A RAG system answers a question. An Agent completes a task — by planning steps, calling tools, observing results, and deciding what to do next. It is the difference between a reference book and a junior engineer.

Three things make an Agent different from a plain LLM or RAG system:

🧠
Reasoning

The agent breaks a complex task into sub-steps, decides which step to take next, and changes course when something doesn't work. It thinks before it acts.

Example: "Summarise this document, find related policies, and draft a compliance report" — the agent figures out the sequence itself.
🔧
Tool Use

Agents call external tools — query a database, run code, search the web, call an API. The LLM decides which tool to use and when.

Example: A Text2SQL agent that receives a natural language question, calls a tool to fetch the schema, generates SQL, validates it, then executes it.
💾
Memory

Agents maintain context across steps — short-term within a task and long-term across sessions. This makes multi-turn stateful workflows possible.

Example: A clinical AI agent that remembers a patient's history across multiple consultations, not just the current query.
RAG System
User asks a question → retrieve context → LLM answers → done. One round trip. Passive.
vs
Agent System
User gives a goal → agent plans → calls tools → observes results → replans → completes task. Multiple loops. Active.

MCP — Model Context Protocol

The USB-C of AI tools — one standard protocol to connect models to anything

As agents got more powerful, a new problem emerged: every tool had its own integration method. Connecting an LLM to a database looked different from connecting it to an API, which looked different from connecting it to a file system. Engineers were writing glue code for every single combination.

MCP, introduced by Anthropic in 2024, solves this with a single open standard. It defines how an AI model discovers and calls external tools — databases, APIs, file systems, code interpreters — using one consistent protocol. Think of it as USB-C for AI tools: one connector that works everywhere.

The agent doesn't need to know how the database works internally. It just calls MCP tools and gets structured results back. This is the same pattern used in production AI systems at scale.

09 · Live Demo

Text2SQL Agent with MCP — Live

A production-pattern agent built with FastMCP, GPT-4.1-mini, and Chainlit — running live

Everything we've discussed today — the stack, the LLM, RAG, agents, MCP — comes together in this demo. This is not a toy. It follows the exact same architectural patterns used in enterprise AI systems. The only difference is the scale.

Stack used in this demo
💬 Chainlit — conversational UI 🧠 GPT-4.1-mini — agent brain ⚙️ FastMCP — tool server 🗄️ SQL Database — data source 🐍 Python — everything
View source on GitHub — github.com/Parvez-Khan-1/text2sql_agent

What happens when you type a question:

01
User types a natural language question

"How many transactions above ₹10,000 happened in Q3?" — Chainlit captures this and sends it to the agent.

02
Agent calls MCP to understand the data

Before writing any SQL, the agent calls get_tables and get_schema via the MCP server. It reads the glossary and ER diagram resources to understand relationships.

03
Agent generates and validates SQL

GPT-4.1-mini generates a SQL query based on the schema. It then calls validate_sql via MCP to check for errors before executing — preventing bad queries from hitting the database.

04
Agent executes and synthesises

execute_sql runs the validated query. Results come back as structured data. The agent synthesises a plain English answer with the numbers embedded — not just a raw table dump.

05
Chainlit renders the response

The final answer streams back into the Chainlit UI — with the SQL shown as a collapsible detail, so users can verify the query that produced their answer.

Why this architecture matters

Notice what the agent never does — it never hardcodes the schema, never assumes table names, never trusts its own SQL without validation. It discovers everything through MCP at runtime. This is what makes it robust enough for a production database where schemas change. The same pattern works for any domain — healthcare records, financial transactions, legal documents.
